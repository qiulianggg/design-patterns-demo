# 装饰器模式（Decorator）

## 意图

> **动态地**给一个对象添加额外的职责。就扩展功能而言，装饰器比继承（子类化）**更灵活**。

## 解决什么问题

想给对象加功能，第一反应可能是「写个子类」。但如果功能可以**任意组合**（大写、加密、压缩、缓冲……），
子类会组合爆炸（`UpperCaseEncryptedBufferedSource`……）。

装饰器把每个附加功能做成一个**包装器**，它实现和被包装对象**相同的接口**，内部持有被包装对象，
在转调的前后插入自己的逻辑。于是功能可以像**套娃**一样在运行时自由叠加。

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| Component | 核心接口 | `DataSource` |
| ConcreteComponent | 被装饰的原始对象 | `PlainSource` |
| Decorator | 持有 Component、实现同一接口的基础装饰器 | `decorator` |
| ConcreteDecorator | 具体附加职责 | `UpperCaseDecorator`、`ExclaimDecorator` |

## 结构

```
Exclaim( Upper( Plain ) )
   │        │       │
   ▼        ▼       ▼
 加"!!!"  转大写   原始数据
调用时从外向内委托，返回时从内向外叠加。
```

## Go 惯用写法

Go 里装饰器靠**内嵌 + 相同接口**实现：装饰器持有 `wrapped DataSource`，
`Write` 里先 `d.wrapped.Write(...)` 再叠加自己的处理。由于接口隐式实现，包装链上每层都是 `DataSource`，可无限嵌套。

一个非常典型的真实例子是标准库 `io`：`bufio.NewReader(gzip.NewReader(file))` ——
`Reader` 层层包装，每层都还是 `io.Reader`，这就是装饰器。HTTP 中间件 `func(http.Handler) http.Handler` 也是装饰器思想。

## 装饰器 vs 相近模式

- **装饰器 vs 继承**：装饰器运行时组合、可叠加；继承编译期固定。
- **装饰器 vs 适配器**：装饰器**不改接口**只增强；适配器**改接口**做兼容。
- **装饰器 vs 代理**：结构几乎一样，但代理侧重**控制访问**（懒加载、权限、远程），装饰器侧重**增强功能**。

## 适用场景

- 需要在不影响其他对象的情况下，给单个对象动态、透明地加功能；
- 功能可自由组合、可叠加、可撤销；
- 无法或不便用继承扩展（类被 final、或组合数量太多）。

## 优点

- 比继承灵活，运行时自由组合；
- 符合单一职责（每个装饰器只做一件事）和开闭原则。

## 缺点

- 多层包装后调试困难，栈很深；
- 装饰器与被装饰对象的初始化顺序/配置容易出错；
- 小功能也要一个类型，数量可能变多。

## 运行

```bash
go run ./structural/decorator
```
