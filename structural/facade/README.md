# 外观模式（Facade）

## 意图

> 为子系统中的一组接口提供一个**统一的高层接口**，使子系统更容易使用。

## 解决什么问题

一个功能背后往往涉及多个子系统的**多步、有顺序**的调用（开投影仪→切宽屏→开功放→调音量→开播放器→播放）。
如果让每个调用方都记住这一整套流程，既繁琐又容易出错，还把调用方和子系统内部**强耦合**。

外观提供一个「一键搞定」的方法 `WatchMovie`，把复杂流程藏在背后。调用方只面对简单接口。

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| Facade | 提供简化的高层接口，编排子系统 | `HomeTheaterFacade` |
| Subsystem | 各自独立的复杂子系统 | `Projector`、`Amplifier`、`Player` |
| Client | 只与 Facade 打交道 | `main` |

## 结构

```
Client ──► HomeTheaterFacade.WatchMovie()
                 │  编排调用
      ┌──────────┼───────────┐
  Projector   Amplifier    Player
```

## Go 惯用写法

Go 里外观就是一个**聚合了若干子系统的结构体**，对外暴露少量语义清晰的方法。
这在 Go 项目里极其常见——比如一个 `Service` 层封装了 repository、cache、mq 等多个依赖，
对 handler 只暴露 `PlaceOrder()` 这样的高层方法，本质就是外观。

要点：**外观不阻止你直接使用子系统**。它只是提供便利入口；需要精细控制时仍可绕过它直接调子系统。

## 外观 vs 相近模式

- **外观 vs 适配器**：外观**定义一个新的简化接口**去统御一堆子系统；适配器是把**一个已有对象**转换成**另一个已有接口**以兼容。
- **外观 vs 中介者**：外观是**单向**简化（客户端→子系统），子系统不知道外观；中介者让同级对象**双向**通过它通信。

## 适用场景

- 想给复杂子系统提供简单入口；
- 想给子系统分层，用外观定义每层的入口，降低层间耦合；
- 封装第三方库/遗留系统，减少直接依赖面。

## 优点

- 简化使用，降低客户端与子系统的耦合；
- 子系统内部变化不影响客户端（只要外观接口稳定）。

## 缺点

- 外观可能演变成「什么都往里塞」的上帝对象；
- 过度封装会隐藏必要的灵活性。

## 运行

```bash
go run ./structural/facade
```
