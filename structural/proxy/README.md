# 代理模式（Proxy）

## 意图

> 为其他对象提供一种**代理（替身）**，以**控制对这个对象的访问**。

## 解决什么问题

有时你不希望（或不能）让客户端直接访问某个对象，而想在访问的前后插入控制逻辑：
延迟创建昂贵对象、权限校验、缓存、记录日志、远程调用、限流等。

代理与真实对象实现**同一个接口**，客户端拿到的是代理，用起来和真实对象一模一样，但访问被代理悄悄「接管」了。

## 常见代理类型

| 类型 | 作用 | 例子 |
|------|------|------|
| 虚拟代理 | 延迟创建开销大的对象 | 本例的图片懒加载 |
| 保护代理 | 访问控制 / 权限校验 | 按角色决定能否调用 |
| 远程代理 | 代表远程对象（RPC 存根） | gRPC/RPC client stub |
| 缓存代理 | 缓存结果，避免重复计算 | HTTP 缓存 |
| 智能引用 | 附加计数、日志等 | 引用计数、访问日志 |

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| Subject | 真实对象与代理的公共接口 | `Image` |
| RealSubject | 真正干活的对象 | `RealImage` |
| Proxy | 持有 RealSubject 引用，控制访问 | `ProxyImage` |

## 结构

```
Client ──► Image(接口)
              ▲
          ProxyImage ──按需创建/控制──► RealImage
        (日志/懒加载/权限)              (真正显示)
```

## Go 惯用写法

Go 里代理同样靠**相同接口 + 组合**：代理结构体实现 `Image`，内部持有 `*RealImage`（可延迟到首次使用才 `new`）。

在真实 Go 项目中，代理思想无处不在：
- `http.RoundTripper` 包一层做重试/鉴权/度量；
- RPC 生成的 client stub 就是远程代理；
- 给 repository 加一层缓存代理。

若代理在并发下懒加载，需用 `sync.Once` 或加锁保护初始化。

## 代理 vs 装饰器（结构几乎相同，意图不同）

- **代理**：控制**是否/如何**访问对象，通常自己管理真实对象的生命周期，接口在编译期就确定要代理谁；
- **装饰器**：为对象**增强/叠加**功能，真实对象由外部传入、可层层堆叠。

## 适用场景

- 对象创建昂贵、希望延迟到真正需要时（虚拟代理）；
- 需要在访问前后加权限、日志、缓存、限流（保护/缓存代理）；
- 访问的是远程对象（远程代理）。

## 优点

- 在不改动真实对象的前提下控制访问，符合开闭原则；
- 可对客户端透明地优化（懒加载、缓存）。

## 缺点

- 增加一层间接，可能引入延迟；
- 类数量增加，逻辑变复杂。

## 运行

```bash
go run ./structural/proxy
```
