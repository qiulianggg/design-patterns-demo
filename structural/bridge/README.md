# 桥接模式（Bridge）

## 意图

> 将**抽象部分**与它的**实现部分**分离，使它们都可以**独立地变化**。

这里的「抽象」和「实现」不是指 Go 的 interface，而是指一个事物的**两个正交的变化维度**。

## 解决什么问题

当一个类存在**两个（或多个）独立变化的维度**时，如果用继承去覆盖所有组合，类会**爆炸式增长**。

本例：消息有「类型」（普通/加急）和「渠道」（短信/邮件）两个维度。
- 用继承：`普通短信`、`加急短信`、`普通邮件`、`加急邮件` = 2×2 = 4 个类；再加一个维度就乘一遍。
- 用桥接：类型 2 个 + 渠道 2 个 = 4 个，但**加法而非乘法**增长，且任意组合。

桥接的核心动作：**用组合（has-a）替代继承（is-a）**，把一个维度抽成接口，让另一维度持有它。

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| Abstraction | 抽象维度，持有 Implementor 引用 | `Message` / `NormalMessage`、`UrgentMessage` |
| Implementor | 实现维度的接口 | `MessageSender` |
| ConcreteImplementor | 实现维度的具体类 | `SMSSender`、`EmailSender` |

## 结构

```
   Abstraction 维度            Implementor 维度
  ┌──────────────┐   桥      ┌──────────────┐
  │ NormalMessage │─────────►│ MessageSender │
  │ UrgentMessage │  持有     └──────────────┘
  └──────────────┘             ▲         ▲
                          SMSSender   EmailSender
```

两个维度各自独立扩展：加一种消息类型不动渠道，加一种渠道不动消息类型。

## Go 惯用写法

Go 里桥接就是「结构体内嵌一个接口字段」：`Message` 内嵌 `sender MessageSender`，
具体消息类型通过内嵌 `Message` 复用这个桥。这是组合优于继承的直接体现。

## 桥接 vs 策略

结构上很像（都持有一个接口），区别在**意图**：
- **策略**：一个维度，运行时替换**算法**；
- **桥接**：两个维度，做**结构解耦**，让抽象与实现分别演化。

## 适用场景

- 类有两个及以上独立变化维度（跨平台 GUI：控件类型 × 操作系统渲染）；
- 想在运行时切换实现；
- 避免抽象与实现之间的永久绑定。

## 优点

- 抽象与实现解耦，各自独立扩展；
- 避免类爆炸；符合开闭原则、单一职责。

## 缺点

- 增加了理解和设计难度；对只有一个维度的简单场景没必要。

## 运行

```bash
go run ./structural/bridge
```
