# 享元模式（Flyweight）

## 意图

> 运用**共享**技术，有效地支持**大量细粒度对象**，节省内存。

## 解决什么问题

当程序需要创建**海量相似对象**（编辑器里的每个字符、游戏里的每棵树/每颗子弹、地图上的每个图标），
如果每个对象都完整保存所有数据，内存会爆炸。但仔细看，这些对象的大部分数据是**重复**的。

享元的核心洞见：把对象状态分成两部分——
- **内部状态（intrinsic）**：与对象本身绑定、可在多个对象间**共享**（字符的字形、字体）；
- **外部状态（extrinsic）**：随使用场景变化、不能共享，由客户端在调用时**传入**（字符在页面上的坐标）。

只为不同的内部状态创建**一份**共享对象，外部状态则在方法调用时传参，从而大幅减少对象数量。

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| Flyweight | 保存内部状态、接收外部状态的对象 | `Glyph` |
| FlyweightFactory | 缓存并复用享元，保证共享 | `GlyphFactory` |
| Client | 持有外部状态，向工厂取享元 | `main` |

## 结构

```
Client ──Get(char,font)──► GlyphFactory
                              pool: map[key]*Glyph  (相同 key 只存一份)
Client ──Render(x,y)─────► Glyph   (x,y 是外部状态，运行时传入)
```

本例渲染 "hello" 共 5 个字符，但 `'l'` 出现两次共享同一个享元，最终只创建 4 个 `Glyph` 对象。

## Go 惯用写法

Go 里享元工厂就是一个 `map` 缓存 + 一个「取不到就新建」的方法。若在并发下使用，需要给 map 加锁（`sync.Mutex`）或用 `sync.Map`。

关键纪律：**享元对象必须是「不可变」的共享数据**，绝不能把外部状态存进去，否则一处修改会影响所有引用它的地方。所以 `Render(x, y)` 把坐标作为**参数**而非字段。

> 注意别把享元和「对象池（如 `sync.Pool`）」混淆：对象池是复用**可变**对象以减少 GC；享元是共享**不可变**内部状态以减少内存占用。

## 适用场景

- 程序中存在大量对象，内存吃紧；
- 对象的大部分状态可外部化、变成可共享的少量内部状态；
- 对象身份不重要（可以用共享实例替代独立实例）。

## 优点

- 大幅节省内存；
- 相同内部状态集中管理。

## 缺点

- 引入内部/外部状态的划分，代码更复杂、可读性下降；
- 用 CPU 时间（每次传外部状态、查缓存）换内存空间。

## 运行

```bash
go run ./structural/flyweight
```
