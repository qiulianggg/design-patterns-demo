# 组合模式（Composite）

## 意图

> 将对象组合成**树形结构**以表示「部分-整体」的层次关系。组合模式使得客户端对**单个对象**
> 和**组合对象**的使用具有**一致性**。

## 解决什么问题

当数据天然是**树形**的（文件系统、组织架构、UI 控件树、菜单），你希望**不区分**「这是一个叶子还是一个容器」，
统一地对它们调用同样的方法（如计算大小、渲染、遍历）。

组合模式让叶子和容器都实现**同一个接口**，容器把操作**递归委托**给它的子节点，客户端代码因此无需写一堆 `if 是目录 else 是文件`。

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| Component | 叶子与容器的统一接口 | `Node` |
| Leaf | 叶子节点，无子节点 | `File` |
| Composite | 容器节点，持有子 Component，递归处理 | `Directory` |
| Client | 统一操作 Component | `main` |

## 结构

```
            Node(接口): Name/Size/Print
              ▲                    ▲
            File(叶子)        Directory(容器)
                               children []Node ──┐
                                    ▲            │
                                    └── 递归包含 ─┘
```

## Go 惯用写法

- 用**接口** `Node` 统一叶子与容器；
- 容器 `Directory` 持有 `children []Node`，其方法（`Size`、`Print`）遍历子节点**递归调用**同名方法；
- `Add` 返回 `*Directory` 以支持链式构建。

Go 没有继承，叶子和容器分别实现接口即可，天然契合组合模式。

## 「透明」与「安全」两种设计

- **透明式**：把 `Add/Remove` 也放进 `Node` 接口——叶子和容器接口完全一致，但叶子得实现无意义的 `Add`。
- **安全式**（本例）：`Add` 只放在 `Directory` 上——更类型安全，但客户端处理子节点时需要类型断言才能 `Add`。

Go 中通常倾向**安全式**，把公共只读操作放接口，把结构修改操作放具体容器类型。

## 适用场景

- 表示部分-整体层次结构（文件树、DOM、组织架构、表达式树）；
- 希望客户端忽略「组合对象」与「单个对象」的差异。

## 优点

- 客户端代码简单统一，无需区分节点类型；
- 新增节点类型容易，符合开闭原则；
- 天然支持递归结构。

## 缺点

- 接口设计要在「透明」与「安全」间权衡；
- 过度一般化会让类型约束变弱（难以限制某容器只能放某种子节点）。

## 运行

```bash
go run ./structural/composite
```
