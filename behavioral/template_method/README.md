# 模板方法模式（Template Method）

## 意图

> 定义一个操作中**算法的骨架**，而将一些步骤延迟到子类中实现。模板方法使得子类可以在
> **不改变算法结构**的情况下，重新定义算法的某些特定步骤。

## 解决什么问题

多个流程的**整体步骤和顺序相同**，只有**个别步骤的具体做法不同**（读取→处理→保存，
只是数据源和处理逻辑不同）。如果每个流程都完整重写一遍，就会有大量重复代码，且骨架容易走样。

模板方法把「不变的骨架」写在一处（固定步骤顺序），把「会变的步骤」抽象出去交给具体实现。
这是**复用算法结构、只替换差异步骤**的经典手段——一种基于「反向控制（好莱坞原则：别调用我，我会调用你）」的复用。

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| AbstractClass | 定义模板方法（骨架）和抽象步骤 | `Run` 函数 + `DataProcessor` 接口 |
| ConcreteClass | 实现可变步骤 | `CSVProcessor`、`JSONProcessor` |

## 结构

```
Run(p)  // 模板方法：骨架固定
 ├─ p.Read()          ← 可变步骤，由实现决定
 ├─ p.Process(raw)    ← 可变步骤
 └─ p.Save(result)    ← 可变步骤
```

## Go 惯用写法

经典模板方法依赖**继承**：父类写模板方法，调用被子类重写的抽象方法。Go 没有继承，有两种地道替代：

### 1. 接口 + 独立模板函数（本例）

把可变步骤定义为接口 `DataProcessor`，把骨架写成一个普通函数 `Run(p DataProcessor)`。
骨架固定、步骤委托——干净且无继承。

### 2. 内嵌结构体 + 函数字段（近似「可重写」）

用一个 base 结构体内嵌进具体类型，把可变步骤设计成函数字段注入。但要注意 Go 内嵌**没有多态**：
base 的方法不会「回调」到外层类型重写的方法（没有虚函数），所以纯内嵌模拟继承式模板方法容易踩坑，**推荐用方式 1**。

> 观察：方式 1 其实和**策略模式**非常接近——都是「把可变部分抽成接口注入」。区别在于模板方法保留了一个
> **固定编排多个步骤的骨架**，而策略通常替换的是**单个完整算法**。

## 适用场景

- 多个流程骨架相同、细节不同，想复用骨架、消除重复；
- 框架希望「控制主流程」，把可定制点开放给使用者（典型的框架回调/钩子）；
- 想集中管控算法结构，禁止子类改变步骤顺序。

## 优点

- 复用公共骨架，消除重复代码；
- 变化点集中、可控，符合开闭原则；
- 实现「好莱坞原则」的反向控制。

## 缺点

- 骨架固定，灵活性受限；
- 步骤过多时子类要实现的方法多；
- Go 中若强行用内嵌模拟继承式重写，会因缺乏多态而出错。

## 运行

```bash
go run ./behavioral/template_method
```
