# 状态模式（State）

## 意图

> 允许一个对象在其**内部状态改变时改变它的行为**，对象看起来好像修改了它所属的类。

## 解决什么问题

当一个对象的行为取决于它的状态，且必须在运行时根据状态改变行为时，
朴素写法是在每个方法里写一堆 `switch state { case ... }`。状态一多、操作一多，
这些散落各处的分支就变得难以维护，加新状态要改所有方法。

状态模式把**每个状态**做成一个类，把「该状态下各操作的行为」和「到下一个状态的转移」都封装在状态类里。
上下文只持有「当前状态」，并把操作**委托**给它。加新状态 = 加一个类，不动老代码。

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| Context | 持有当前状态，委托操作 | `Order` |
| State | 状态接口，声明各操作 | `State` |
| ConcreteState | 具体状态的行为 + 状态转移 | `PendingState`、`PaidState`、`ShippedState` |

## 结构

```
Order(Context).state ──► 当前 State
  Pay()  ─委托─► state.Pay(o)  ──► 可能 o.setState(下一个状态)
  Ship() ─委托─► state.Ship(o)
待支付 ──Pay──► 已支付 ──Ship──► 已发货
```

## Go 惯用写法

Go 里每个状态是实现 `State` 接口的结构体（通常无字段，空结构体）。状态转移由状态自己调用 `o.setState(...)` 完成。上下文的方法只是「转发给当前状态」。

对于**简单**的状态机，Go 里也常直接用**常量 + `switch`** 或状态转移表（`map[state]map[event]state`）来实现，更轻量。当每个状态的行为差异大、逻辑复杂时，才用状态模式的「一状态一类型」。

## 状态 vs 策略（结构几乎相同）

两者都是「上下文持有一个接口、把行为委托出去」，区别在意图：

- **策略**：由**客户端**主动选择/替换算法；各策略之间**互不知道**、也不引起彼此切换。
- **状态**：状态转移通常由**状态自己或上下文内部**驱动；各状态**知道**并会切换到其他状态，构成一台状态机。

## 适用场景

- 对象行为随状态变化，且状态较多、转移规则明确（订单、连接、审批、游戏角色）；
- 想消除大量与状态相关的条件分支。

## 优点

- 消除臃肿的条件分支，行为按状态内聚；
- 状态转移显式化，易于理解和扩展（开闭原则）。

## 缺点

- 状态少时引入过多类，反而繁琐；
- 状态转移逻辑分散在各状态类中，需通读才能看全整张状态图。

## 运行

```bash
go run ./behavioral/state
```
