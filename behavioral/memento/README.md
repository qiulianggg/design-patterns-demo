# 备忘录模式（Memento）

## 意图

> 在**不破坏封装**的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，
> 以便以后将对象**恢复**到原先保存的状态。

## 解决什么问题

要实现「撤销/回滚/存档」，就得保存对象某一时刻的状态，之后能还原回去。
但如果为了保存状态而把对象的内部字段全部暴露成 public，就破坏了封装。

备忘录模式提供一个「快照对象」：由对象自己（Originator）生成和读取，第三方（Caretaker）只负责保管这些快照，看不到、也不改内部细节。

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| Originator | 拥有状态，能生成/恢复备忘录 | `Editor` |
| Memento | 状态快照，对外不透明 | `Memento` |
| Caretaker | 保管备忘录，不解读其内容 | `History` |

## 结构

```
Editor.Save()   ──► Memento(content 快照) ──► History.Push()
Editor.Restore(History.Pop())  ◄── 取回快照恢复状态
Caretaker(History) 只存取 Memento，从不读它的内部。
```

## Go 惯用写法

Go 没有 Java 那种严格的访问控制来对不同类隐藏字段，但可以用**包级私有**近似实现「封装」：

- 把 `Memento` 的字段设为**小写（不可导出）**，只有同包的 `Editor` 能读写；
- `History` 即使拿着 `*Memento` 也无法访问其字段，只能整体保管——达到「不破坏封装」的效果。

要点：`Save` 时要注意**深拷贝**含引用类型的状态（同原型模式的坑），否则快照会随对象一起变化。

## 备忘录 vs 命令

- **备忘录**：保存**状态快照**，靠「恢复到某个快照」实现撤销——适合状态大、操作难以逆运算的情况。
- **命令**：保存**操作**并提供 `Undo` 逆操作实现撤销——适合操作可逆、状态大不宜整存的情况。
- 两者常结合：命令历史 + 每步的备忘录。

## 适用场景

- 撤销/重做、事务回滚；
- 游戏存档 / 检查点；
- 需要保存和恢复对象状态，且不愿暴露其内部结构。

## 优点

- 不破坏封装即可保存/恢复状态；
- 简化 Originator（保管职责交给 Caretaker）。

## 缺点

- 频繁、大量快照消耗内存；
- 生成/深拷贝快照可能开销大；
- Caretaker 需管理快照的生命周期。

## 运行

```bash
go run ./behavioral/memento
```
