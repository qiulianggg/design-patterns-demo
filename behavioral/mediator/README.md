# 中介者模式（Mediator）

## 意图

> 用一个**中介对象**来封装一系列对象之间的交互。中介者使各对象**不需要显式地相互引用**，
> 从而使其耦合松散，且可以独立地改变它们之间的交互。

## 解决什么问题

当一组对象**两两之间都要通信**时，会形成「网状」的相互引用（n 个对象最多 n² 条关系），
牵一发而动全身，难以维护和复用。

中介者把这些交互集中到一个对象里，网状结构变成**星形**：每个对象只和中介者通信，中介者负责协调转发。

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| Mediator | 中介者接口 | `Mediator` |
| ConcreteMediator | 协调各同事的交互 | `ChatRoom` |
| Colleague | 同事，只认识中介者 | `User` |

## 结构

```
网状(耦合高)              星形(经中介者)
A─B─C                      A   B
│╳│  ...                    \ /
D─E                     ┌── ChatRoom ──┐
                        C     D      E
每个 User 只和 ChatRoom 通信，彼此不直接引用。
```

## Go 惯用写法

Go 里中介者就是一个持有所有同事引用、提供协调方法的结构体。同事通过接口 `Mediator` 依赖它，避免与具体中介者耦合。

要点：中介者容易膨胀成「上帝对象」，把所有逻辑都吸进去。控制它的职责边界很重要。Go 中也常用 **channel + 一个协调 goroutine** 来实现中介者式的消息分发（如聊天室、事件总线）。

## 中介者 vs 相近模式

- **中介者 vs 外观**：外观是**单向**、只简化对子系统的调用，子系统不知道外观；中介者是**多向**协调，同事们主动依赖它、通过它互相通信。
- **中介者 vs 观察者**：常配合使用。观察者是「一对多的通知广播」；中介者是「多对多交互的集中协调」。中介者内部可以用观察者来实现通知。

## 适用场景

- 一组对象通信复杂、相互引用混乱；
- 想复用某个对象，但它引用了太多其他对象；
- GUI 组件协调、聊天室、航空管制、工作流引擎。

## 优点

- 降低对象间耦合，网状变星形；
- 交互逻辑集中，易于修改和复用同事类；符合单一职责。

## 缺点

- 中介者可能变得庞大复杂，成为新的维护难点（上帝对象）。

## 运行

```bash
go run ./behavioral/mediator
```
