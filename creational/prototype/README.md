# 原型模式（Prototype）

## 意图

> 用**原型实例**指定要创建对象的种类，并通过**拷贝（克隆）这个原型**来创建新对象。

## 解决什么问题

有时创建一个对象成本很高（要查数据库、复杂计算、大量初始化），或者你想基于一个「模板对象」
快速生成许多相似副本。此时与其从零 `new` 再逐字段赋值，不如直接**复制一个现成的对象**。

原型模式也让「创建对象」与「对象的具体类型」解耦——拿着一个 `Cloneable` 就能复制它，不必知道它到底是什么类型。

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| Prototype | 声明克隆方法的接口 | `Cloneable` |
| ConcretePrototype | 实现克隆 | `Document` |
| Client | 通过调用 `Clone()` 得到新对象 | `main` |

## 深拷贝 vs 浅拷贝（Go 的关键坑）

这是原型模式在 Go 里**最容易出错的地方**：

- Go 的结构体赋值 `b := *a` 是**逐字段值拷贝**；
- 但对于**引用类型字段**（slice、map、指针、channel），拷贝的只是「引用/表头」，
  底层数据仍然**共享**。修改副本会污染原型！

所以 `Clone()` 必须对引用类型字段做**深拷贝**：

```go
tags := make([]string, len(d.Tags))
copy(tags, d.Tags)          // 复制切片内容
// map 同理需要 make + 逐项复制；嵌套指针要递归 Clone
```

本例特意用 `Tags []string` 演示：正确复制后，向副本 `append` 不会影响原型。

> 需要「无脑深拷贝」时，也有人用 `encoding/gob` 或 `json` 序列化再反序列化实现，
> 简单但有性能开销、且无法拷贝不可导出字段/函数。

## 适用场景

- 对象创建成本高，复制比新建更划算；
- 需要基于模板批量生成相似对象（配置模板、游戏中的敌人预制体）；
- 想在运行时通过复制已有实例来决定新对象类型，减少子类。

## 优点

- 避免重复的昂贵初始化；
- 运行时增删原型很灵活，无需绑定具体类；
- 可以隐藏创建的复杂性。

## 缺点

- 深拷贝含循环引用的复杂对象图时实现麻烦；
- 每个类都要实现 `Clone`，且要小心引用字段。

## 运行

```bash
go run ./creational/prototype
```
