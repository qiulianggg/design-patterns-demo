# 单例模式（Singleton）

## 意图

> 保证一个类只有**一个实例**，并提供一个**全局访问点**来获取它。

## 解决什么问题

有些对象在整个程序生命周期内应当**只存在一份**，例如：配置中心、连接池、日志器、
ID 生成器。如果随意 `new`，会导致状态不一致、资源浪费（比如多个数据库连接池）。

单例模式把「只创建一个」和「全局可访问」这两件事封装起来，交给类型自己保证。

## 角色

| 角色 | 说明 | 本例对应 |
|------|------|---------|
| Singleton | 持有唯一实例的类型，并提供获取入口 | `config` + `GetConfig()` |

## 结构

```
GetConfig() ──► sync.Once.Do(初始化) ──► 唯一的 *config
     ▲                                        │
     └────────── 后续调用直接返回 ◄────────────┘
```

## Go 惯用写法

Go 里实现单例最地道的方式是 **`sync.Once`**：

```go
var (
    instance *config
    once     sync.Once
)

func GetConfig() *config {
    once.Do(func() { instance = &config{...} })
    return instance
}
```

`once.Do` 保证传入的函数在多 goroutine 并发下**有且仅执行一次**，天然线程安全，
比手写「双重检查锁定（DCL）」更简洁、也不会出错。

### 其他几种写法对比

1. **包级变量 + `init()`**（饿汉式）：程序启动即创建，最简单，适合初始化不昂贵、必然会用到的场景。
   ```go
   var instance = &config{...} // 或在 init() 中赋值
   ```
2. **`sync.Once`（懒汉式）**：首次使用时才创建，适合初始化开销大或不一定用到的场景。本例采用。
3. **加锁的 DCL**：Go 中**不推荐**，容易写错，`sync.Once` 已经覆盖该场景。

## 适用场景

- 全局配置、日志、监控上报器；
- 数据库连接池、HTTP 客户端等**创建开销大**的共享资源；
- 需要全局唯一协调点的组件（如全局 ID 生成器）。

## 优点

- 保证唯一实例，节约资源；
- 提供全局访问点，使用方便。

## 缺点 / 注意事项

- **是一种全局状态**，会增加耦合、给单元测试带来困难（难以替换 mock）。Go 社区更推荐
  「依赖注入」：把依赖作为参数显式传入，而不是到处 `GetXxx()`。
- 单例若持有可变状态，需自己保证并发安全（本例的 `Set` 在真实场景应加锁）。
- 违反单一职责原则（既管理生命周期又提供业务功能）。

## 运行

```bash
go run ./creational/singleton
```

输出会显示「初始化」只打印一次，且 `c1 == c2`，通过任一引用修改都能被另一个看到。
